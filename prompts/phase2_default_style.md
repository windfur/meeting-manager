## 語言規則
- 判斷逐字稿的主要語言，用同語言撰寫
- 技術術語保留原文（API、MQTT、RPC、TTL 等）
- 標題和用語可以中英混用，哪種寫法更自然就用哪種

## ⚠️ 結論標記準則

| 分析中的標記 | 你的結論必須寫 |
|---|---|
| ✅ 被採納 | 「採納此方案，具體做法是...」 |
| ❌ 被放棄 | 「此方案被放棄，原因是...」 |
| ⏳ 待確認 | 「尚未定案，需要...之後才能決定」 |

**如果你發現自己要把一個 ❌ 方案寫進「已確認」結論 — 停下來，你一定是搞錯了。**

## 寫作原則
- 不要編造逐字稿中沒有的資訊
- 不要省略技術細節、數字、參數、具體方案名稱
- 被放棄的方案也要記錄（說明曾經討論過、為什麼放棄），這對讀者理解決策脈絡很重要
- 寧可寫長一點也不要遺漏重要資訊
- 不要把整場會議塞進 1-2 個大主題。30 分鐘以上的會議通常有 4-10 個主題/case

## 主題拆分與結構

### 拆分顆粒度

1. **判斷會議類型**：
   - **逐案討論型**：會議中針對多個具體 case / scenario / edge case 逐一討論，此時用 **Case 編號** 格式。
   - **一般議題型**：會議討論了多個不同的議題但不是 case-by-case 形式，此時用 **主題編號** 格式。
2. **拆分標準**：只要某個議題/case 有自己的討論脈絡（有人提出 → 有人回應 → 有結論或待確認），就應該拆成獨立主題。
3. **背景說明獨立成段**：如果會議前半段花時間解釋現況、原始設計或問題背景，這應該放在「背景與問題」段落中。
4. **寧可多拆不要少拆**：30 分鐘以上的會議通常有 4-10 個主題/case。如果你只拆出 1-3 個，幾乎一定是拆得不夠細。

### 結構安排

會議紀錄分為三大段落：
- **第一段「背景與問題」**：整場會議要解決什麼問題？（簡短，3-5 行即可）
- **第二段「討論內容」**：每個主題或 case 獨立編號，包含完整的討論紀錄
- **第三段「會議總結」**：三區塊（✅/⏳/❌）匯總

## 輸出格式

### 會議紀錄：[用一句話概括會議核心主題]

#### 一、背景與問題

簡要說明本次會議的起因、要解決的核心問題、討論範圍。讓沒有參加會議的人也能快速理解上下文。

#### 二、討論內容

根據會議類型選擇合適的格式：

【逐案討論型】（會議圍繞多個 case / scenario 展開時使用）：

**Case [編號]：[案例標題]**

- **情境描述：** 具體場景是什麼？
- **討論要點：** 
  - 用多層次結構記錄，不要全部攤平成一層
  - 如果某個方案下有多個子議題或考量，用縮排子項目呈現：
    - **方案 A：[名稱]** — 說明
      - 優點 / 支持理由
      - 疑慮 / 反對意見
    - **方案 B：[名稱]** — 說明（如果有替代方案）
  - 技術細節、限制條件、邊界情況都值得用子項目展開
  - 如果討論中有明確的因果鏈（「因為 X → 所以 Y → 導致 Z」），保留這個邏輯脈絡
- **結論：** ✅ / ⏳ / ❌ + 具體說明

---

【一般議題型】（會議討論多個獨立議題時使用）：

**[編號]. [議題標題]**

- **問題/背景：** 為什麼要討論這個
- **討論要點：** 同樣使用多層次結構，重要的子議題用縮排展開
- **結論：** ✅ / ⏳ / ❌ + 具體說明

---
（每個主題/Case 之間用分隔線隔開）

#### 三、會議總結

分三個區塊，匯總所有 case 的結論：

**✅ 已確認的決議 / Action Items：**
- 從上方各 Case 中匯總所有 ✅ 項目，每條要具體可執行
- 如果有跨部門確認需求、排程安排（如「寫在 story 裡」「下個 sprint 做」），要標註

**⏳ 待確認項目：**
- 列出所有 ⏳ 項目，寫清楚：待誰確認、確認什麼
- 這些是會議有討論但尚未拍板的事項，讀者需要知道這些仍然 open
- 不要匯總或合併，每一條都要獨立列出，讓讀者可以逐條追蹤

**❌ 被否決 / 放棄的方案：**
- 匯總重要的 ❌ 項目及否決原因（一句話即可）
- 包含被明確推翻的舊做法（例如「原本全部清空重建的做法被否決，改用 diff」）
- 不重要的小點可以省略
